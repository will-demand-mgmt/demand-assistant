AI Agent Interface

The single conversational entry point. Understands demand vocabulary, infers intent, and always cites sources.

  • 1
    Deployment channel: Microsoft Teams bot (via Azure Bot Service) is the recommended primary channel. A web widget serves as secondary for external stakeholders.
  • 2
    LLM model: Azure OpenAI (GPT-4o) if on Azure, or Claude API. The system prompt is injected with demand vocabulary and status lifecycle definitions.
  • 3
    Identity & SSO: The agent authenticates via Azure AD. Role is inferred from AD group membership — no separate user management required.
  • 4
    Response format: Structured answers with summary, source citation, and follow-up suggestion. Teams adaptive cards for rich formatting; plain text fallback otherwise.
  • ?
    Teams bot vs web app: Do most stakeholders prefer Teams or a dedicated portal? This drives the primary deployment choice.
  • ?
    External access: Should external stakeholders be able to query the agent? If yes, a guest access model is needed alongside AD integration.
  • ?
    Conversation memory: Stateful (remembers previous questions) or stateless (every query independent)? Stateful enables follow-ups but requires session storage.
  • ?
    Escalation path: When confidence is low, escalate to a demand manager via Teams, or respond "I don't have reliable information on this"?

  • 1
    Embedding model: text-embedding-3-small (OpenAI). Each demand record, document chunk (~512 tokens), and meeting summary is embedded and stored at index time.
  • 2
    Retrieval strategy: Hybrid search — semantic (cosine similarity) + keyword (BM25). Results are re-ranked before passing to the LLM.
  • 3
    Context budget: Top-k=5 chunks injected into LLM context, each tagged with source system and timestamp for accurate citation.
  • 4
    Hallucination guard: LLM instructed to answer only from provided context and say "I don't have this information" when uncertain.
  • ?
    Vector DB choice: Azure AI Search (zero infra if already on Azure) vs Pinecone vs Weaviate. Depends on your cloud environment.
  • ?
    Chunk size: 512 tokens default. Demand records with long requirement fields may need larger windows. Needs empirical testing with real A-ah! data.
  • ?
    Multi-lingual support: Are demands and documents in multiple languages? If yes, a multilingual embedding model is needed from day one.
  • ?
    Re-ranking latency: A cross-encoder adds ~100–200ms. Acceptable trade-off for your use case?

Intent Router

Classifies every query and dispatches it to the right connector — or multiple connectors in parallel.

  • 1
    Status query → A-ah! connector. Keywords: demand ID (DM-###), "status", "stage", "phase", "shaping", "design", "roll-out".
  • 2
    Document query → SharePoint connector. Keywords: "document", "brief", "deliverable", "presentation", "PDF", "latest version".
  • 3
    Meeting query → Teams connector. Keywords: "meeting", "minutes", "decision", "discussed", "action item", date references.
  • 4
    Dev query → Jira connector. Keywords: "ticket", "sprint", "blocker", "developer", "progress", "release".
  • 5
    Multi-source → parallel dispatch. "Status of DM-044 and last meeting decision?" triggers A-ah! + Teams simultaneously.
  • ?
    Classification approach: LLM zero-shot (simpler) or fine-tuned classifier (faster at scale)? Query volume drives the decision.
  • ?
    Ambiguous queries: Ask a clarifying question, or retrieve from all sources and let the synthesizer decide?
  • ?
    Fallback: If no connector matches, broad knowledge store search or a clear "no source found" response?
