AI Agent Interface

The single conversational entry point. Understands demand vocabulary, infers intent, and always cites sources.

  • 1
    Deployment channel: Microsoft Teams bot (via Azure Bot Service) is the recommended primary channel — your team already lives in Teams. A web widget serves as secondary for external stakeholders.
  • 2
    LLM model: Azure OpenAI (GPT-4o) if on Azure, or Claude API. The system prompt is injected with demand management vocabulary and your full status lifecycle definitions (Shaping → Closure).
  • 3
    Identity & SSO: The agent authenticates the caller via Azure AD. Role is inferred from AD group membership — no separate user management required.
  • 4
    Response format: Structured answers with a summary, source citation ("Per A-ah! updated 14 May"), and a follow-up suggestion. Teams adaptive cards for rich formatting; plain text fallback otherwise.
  • ?
    Teams bot vs web app: Do most stakeholders prefer Teams or a dedicated portal? This drives the primary deployment choice and development effort.
  • ?
    External access: Should external stakeholders (vendors, clients) be able to query the agent? If yes, a guest access model is needed alongside the AD integration.
  • ?
    Conversation memory: Should the agent remember previous questions within a session (stateful) or treat every query independently? Stateful enables follow-ups but requires session storage.
  • ?
    Escalation path: When confidence is low, should the agent escalate to a demand manager via Teams, or simply respond "I don't have reliable information on this"?

  • 1
    Embedding model: text-embedding-3-small (OpenAI) or equivalent. Each demand record, document chunk (~512 tokens), and meeting summary is embedded and stored in the vector DB at index time.
  • 2
    Retrieval strategy: Hybrid search — semantic (cosine similarity) + keyword (BM25). Semantic catches conceptual matches; keyword catches exact demand IDs. Results are re-ranked before passing to the LLM.
  • 3
    Context window budget: Top-k=5 chunks injected into the LLM context. Each chunk is tagged with its source system and timestamp so the LLM can cite accurately.
  • 4
    Hallucination guard: The LLM is instructed to answer only from provided context and to say "I don't have this information" when uncertain. Source citations in every response make grounding auditable.
  • ?
    Vector DB choice: Azure AI Search (zero infra if already on Azure) vs Pinecone (managed, fast) vs Weaviate (open source). Depends on your cloud environment.
  • ?
    Chunk size: 512 tokens is a common default. Demand records with long requirement fields may need a larger window. Requires empirical testing with real A-ah! data.
  • ?
    Multi-lingual support: Are demands and documents in multiple languages? If yes, a multilingual embedding model (e.g. multilingual-e5) is needed from day one.
  • ?
    Re-ranking latency: A cross-encoder re-ranker improves precision significantly but adds ~100–200ms latency. Acceptable trade-off for your use case?

Intent Router

Classifies every query and dispatches it to the right connector — or multiple connectors in parallel.

  • 1
    Status query → A-ah! connector. Triggered by: demand ID pattern (DM-###), keywords "status", "stage", "phase", "shaping", "design", "roll-out".
  • 2
    Document query → SharePoint connector. Triggered by: keywords "document", "brief", "deliverable", "presentation", "PDF", "latest version", "attached".
  • 3
    Meeting / decision query → Teams connector. Triggered by: keywords "meeting", "minutes", "decision", "discussed", "action item", plus date references.
  • 4
    Development query → Jira connector. Triggered by: keywords "ticket", "sprint", "blocker", "developer", "progress", "release", "fix".
  • 5
    Multi-source query → parallel dispatch. Example: "What is the status of DM-044 and what was decided in the last meeting?" triggers A-ah! + Teams simultaneously and merges results before synthesis.
  • ?
    Classification approach: Use the LLM itself (zero-shot, simpler to maintain) or a lightweight fine-tuned classifier (faster, cheaper at scale)? Volume of queries will determine the right trade-off.
  • ?
    Ambiguous queries: When intent is unclear, should the agent ask a clarifying question or retrieve from all sources and let the synthesizer decide? Clarifying is more accurate; broad retrieval is faster.
  • ?
    Fallback behaviour: If no connector matches, should the agent search the full knowledge store broadly, or return a clear "I couldn't find a relevant source" message?
