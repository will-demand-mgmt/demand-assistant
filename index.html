AI Agent Interface

The single conversational entry point. Understands demand vocabulary, infers intent, and always cites sources.

  • 1
    Deployment channel: Microsoft Teams bot (via Azure Bot Service) is the recommended primary channel. A web widget serves as secondary for external stakeholders.
  • 2
    LLM model: Azure OpenAI (GPT-4o) if on Azure, or Claude API. The system prompt is injected with demand vocabulary and status lifecycle definitions.
  • 3
    Identity & SSO: The agent authenticates via Azure AD. Role is inferred from AD group membership — no separate user management required.
  • 4
    Response format: Structured answers with summary, source citation, and follow-up suggestion. Teams adaptive cards for rich formatting; plain text fallback otherwise.
  • ?
    Teams bot vs web app: Do most stakeholders prefer Teams or a dedicated portal? This drives the primary deployment choice.
  • ?
    External access: Should external stakeholders be able to query the agent? If yes, a guest access model is needed alongside AD integration.
  • ?
    Conversation memory: Stateful (remembers previous questions) or stateless (every query independent)? Stateful enables follow-ups but requires session storage.
  • ?
    Escalation path: When confidence is low, escalate to a demand manager via Teams, or respond "I don't have reliable information on this"?

  • 1
    Embedding model: text-embedding-3-small (OpenAI). Each demand record, document chunk (~512 tokens), and meeting summary is embedded and stored at index time.
  • 2
    Retrieval strategy: Hybrid search — semantic (cosine similarity) + keyword (BM25). Results are re-ranked before passing to the LLM.
  • 3
    Context budget: Top-k=5 chunks injected into LLM context, each tagged with source system and timestamp for accurate citation.
  • 4
    Hallucination guard: LLM instructed to answer only from provided context and say "I don't have this information" when uncertain.

  • ?
    Re-ranking latency: A cross-encoder adds ~100–200ms. Acceptable trade-off for your use case?

Intent Router

Classifies every query and dispatches it to the right connector — or multiple connectors in parallel.

  • 1
    Status query → A-ah! connector. Keywords: demand ID (DM-###), "status", "stage", "phase", "shaping", "design", "roll-out".
  • 2
    Document query → SharePoint connector. Keywords: "document", "brief", "deliverable", "presentation", "PDF", "latest version".
  • 3
    Meeting query → Teams connector. Keywords: "meeting", "minutes", "decision", "discussed", "action item", date references.
  • 4
    Dev query → Jira connector. Keywords: "ticket", "sprint", "blocker", "developer", "progress", "release".
  • 5
    Multi-source → parallel dispatch. "Status of DM-044 and last meeting decision?" triggers A-ah! + Teams simultaneously.
  • ?
    Classification approach: LLM zero-shot (simpler) or fine-tuned classifier (faster at scale)? Query volume drives the decision.
  • ?
    Ambiguous queries: Ask a clarifying question, or retrieve from all sources and let the synthesizer decide?
  • ?
    Fallback: If no connector matches, broad knowledge store search or a clear "no source found" response?

Vector Database

The semantic search engine at the heart of the knowledge store. Finds relevant demand knowledge even when the exact words don't match.

Query embedding
"Which demands are related to data governance?"
Scanning vector space…
DM-044 — Data Platform Migration (business_requirements chunk)0.91top match
DM-031 — Data Quality Framework (mission brief)0.87top match
Teams minutes 02/04 — governance discussion action items0.83match
DM-019 — GDPR Compliance Demand (SharePoint brief)0.79match
DM-007 — Legacy System Decommission0.44below threshold
  • 1
    What gets embedded: Each demand record (chunked by field), document section (~512 tokens), and meeting minute summary. Every chunk carries metadata: source system, demand ID, timestamp, and document type.
  • 2
    Embedding model: text-embedding-3-small (OpenAI, 1536 dimensions) for English-primary content. If multilingual content is needed, switch to multilingual-e5-large before first indexing — migration is costly afterwards.
  • 3
    Similarity threshold: Chunks scoring below 0.70 cosine similarity are excluded from retrieval. This prevents loosely related content from polluting the LLM context.
  • 4
    Hybrid search: Semantic vector search is combined with BM25 keyword search. Final ranking = 0.6 × semantic score + 0.4 × keyword score. Exact demand IDs (e.g. DM-044) always score highest via keyword boost.
  • ?
    Vector DB platform: Azure AI Search (best if already on Azure — integrated auth, no extra infra), Pinecone (managed, fastest queries), or Weaviate (open source, full control). What is your cloud environment?
  • ?
    Index size estimate: How many demands exist today, and what is the average length of business requirements? This determines initial storage and embedding cost.
  • ?
    Multilingual content: Are demands or documents written in multiple languages? The embedding model choice must be made before any data is indexed.
  • ?
    Score threshold tuning: The 0.70 threshold is a starting point. It must be calibrated on real demand data — too high and the agent says "I don't know" too often; too low and answers become noisy.

Two-tier retrieval strategy: the agent always checks the metadata cache first for structured lookups (<10ms). Only if the query requires semantic understanding does it hit the vector DB (~80–150ms). This keeps simple factual queries fast and cheap.
  • ?
    Cache technology: Redis (fastest, in-memory) for hot data, PostgreSQL for structured relational queries, or Azure Table Storage if already on Azure. Choice depends on query patterns and existing infra.
  • ?
    Cache TTL: How stale can the metadata be? 15-minute TTL is a safe default. If A-ah! supports webhooks, the cache updates on every status change instead of on a schedule.
  • ?
    Portfolio queries: Should the cache support aggregate queries like "how many demands are in Design phase?" — or is that always delegated to the vector DB? Aggregates are faster from a structured cache.

Access Control Layer

Ensures every user sees only what they are authorized to see — enforced at retrieval time, before any data reaches the LLM.

Executive All demands · All documents · All meeting minutes · Portfolio dashboards full read
Demand Manager All demands · All documents · Meeting minutes for owned demands full read
Developer / PM Assigned demands · Technical docs · Sprint data · No financial data scoped read
Business Stakeholder Own demands only · Public documents · No meeting minutes restricted
  • 1
    Enforcement point: Access control is applied at retrieval time — the vector DB and metadata cache queries are filtered by the caller's role before results are returned. The LLM never sees unauthorized data.
  • 2
    Implementation: Each indexed chunk carries an "allowed_roles" metadata field set at index time (inherited from the source system's permissions). Vector DB filters by this field before returning results.
  • 3
    Role source: Roles are read from Azure AD group membership at query time — no separate user management. Adding a user to the "Demand Manager" AD group immediately grants the corresponding access.
  • 4
    Audit logging: Every query is logged with: caller identity, role, query text, chunks retrieved, and response generated. This log supports compliance review and helps tune the access model over time.
  • ?
    Role granularity: Are the four roles above the right model for your organization, or do you need sub-roles (e.g. Architect, Business Analyst) with different access scopes?
  • ?
    Document-level permissions: Should some SharePoint documents be restricted beyond role level (e.g. confidential financial data visible only to specific named individuals)?
  • ?
    Audit retention: How long should query logs be retained? GDPR may impose constraints on logs containing personal data (query text may reference names or demand details).
  • ?
    Compliance review: Does your organization require a formal Data Protection Impact Assessment (DPIA) before deploying an AI system that indexes internal documents and conversation history?
